# 车载网络安全

## 1. 车载网络安全概述

### 1.1 为什么智能汽车需要网络安全

```
传统汽车 vs 智能汽车攻击面:

传统汽车攻击面 (有限)          智能汽车攻击面 (广阔)
┌──────────────────┐          ┌──────────────────┐
│ · 物理接触        │          │ · 物理接触        │
│   OBD接口        │          │   OBD接口        │
│   诊断端口       │          │   诊断端口       │
│                  │          │                  │
│                  │          │ · 远程攻击入口    │ ← 新增
│                  │          │   T-Box蜂窝网络  │
│                  │          │   WiFi/蓝牙      │
│                  │          │   车机APP        │
│                  │          │                  │
│                  │          │ · V2X通信        │ ← 新增
│                  │          │   RSU路侧单元    │
│                  │          │   V2V车车通信    │
│                  │          │                  │
│                  │          │ · 供应链攻击      │ ← 新增
│                  │          │   第三方软件     │
│                  │          │   外部服务      │
└──────────────────┘          └──────────────────┘

攻击面扩大: 10倍+
风险等级: 从财产损失 → 生命安全
```

### 1.2 网络安全威胁类型

| 威胁类型 | 描述 | 攻击向量 | 危害等级 |
|---------|------|---------|---------|
| **远程控制** | 远程操控车辆功能 | T-Box、云端 | 致命 |
| **数据窃取** | 窃取车辆/用户数据 | 通信监听 | 高 |
| **勒索软件** | 加密车辆系统索赎金 | 软件漏洞 | 高 |
| **服务拒绝** | 使车辆功能失效 | 网络攻击 | 中高 |
| **隐私泄露** | 定位/行为数据泄露 | 数据拦截 | 中 |
| **固件篡改** | 恶意修改ECU固件 | OTA/诊断 | 高 |

### 1.3 典型攻击案例分析

```
历史重大汽车网络安全事件:

2015年 克莱斯勒Jeep事件
├─ 攻击方式: 通过娱乐系统漏洞入侵CAN总线
├─ 影响: 140万辆车召回
├─ 后果: 可远程控制转向、制动
└─ 意义: 汽车行业网络安全觉醒

2016年 特斯拉Model S被入侵
├─ 攻击方式: 通过WiFi和浏览器漏洞
├─ 影响: 车辆制动系统被控制
├─ 修复: 特斯拉OTA快速修复
└─ 意义: OTA安全更新的重要性

2020+ 多起车联网攻击
├─ 攻击方式: 云服务API漏洞、APP漏洞
├─ 影响: 车辆定位泄露、远程控制
└─ 趋势: 攻击手段日益专业化

教训:
├─ 网络安全是功能安全的延伸
├─ 安全需要全生命周期考虑
└─ 快速响应能力至关重要
```

## 2. ISO/SAE 21434 标准

### 2.1 标准概述

ISO/SAE 21434 《Road vehicles — Cybersecurity engineering》是汽车网络安全领域的国际标准，于2021年发布。

```
ISO/SAE 21434 标准框架:

┌─────────────────────────────────────────────────────────┐
│               ISO/SAE 21434 网络安全工程                 │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐ │
│  │ 组织层面     │    │ 项目层面     │    │ 运营层面     │ │
│  │ 网络安全治理 │    │ 产品开发    │    │ 生产/运维   │ │
│  │             │    │             │    │             │ │
│  │ · 管理体系  │    │ · 需求分析  │    │ · 生产控制  │ │
│  │ · 风险评估  │    │ · 设计开发  │    │ · 漏洞管理  │ │
│  │ · 安全文化  │    │ · 测试验证  │    │ · 事件响应  │ │
│  └─────────────┘    └─────────────┘    └─────────────┘ │
│                                                         │
│  全生命周期覆盖: 概念 → 开发 → 生产 → 运维 → 退役      │
└─────────────────────────────────────────────────────────┘

与ISO 26262关系:
├─ ISO 26262: 功能安全 (随机/系统性故障)
├─ ISO 21434: 网络安全 (恶意攻击)
├─ 两者并行，相互补充
└─ 部分重叠 (如安全机制设计)
```

### 2.2 风险管理流程 (TARA)

| 步骤 | 活动 | 输出 |
|-----|------|------|
| **资产识别** | 识别需保护的资产 | 资产清单 |
| **威胁场景** | 定义攻击场景 | 威胁列表 |
| **影响评级** | 评估安全/财务/运营/隐私影响 | 影响等级 |
| **攻击路径** | 分析攻击可行性 | 攻击路径图 |
| **风险评级** | 综合评估风险值 | 风险等级 |
| **处置决策** | 选择风险处置方式 | 处置计划 |

### 2.3 网络安全等级

```
网络安全影响评级 (Security Level):

严重性(Severity) × 攻击可行性(Attack Feasibility) = 风险等级

严重性等级:
├─ QM: 质量管理，无特殊安全要求
├─ SL 1: 低影响，可接受风险
├─ SL 2: 中等影响，需要保护
├─ SL 3: 高影响，强保护要求
└─ SL 4: 极高影响，最高保护

攻击可行性等级:
├─ 高: 无需专业知识，标准工具
├─ 中: 需要一定专业知识和工具
└─ 低: 需要高深专业知识和定制化工具

风险矩阵示例:
          攻击可行性
          高    中    低
       ┌───────────────────┐
严  高 │ SL1  │ SL2  │ SL3 │
重     ├──────┼──────┼──────┤
性  中 │ QM   │ SL1  │ SL2 │
     ├──────┼──────┼──────┤
   低 │ QM   │ QM   │ SL1 │
       └───────────────────┘
```

## 4. 关键安全技术

### 4.1 安全启动 (Secure Boot)

```
安全启动流程:

┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  上电    │───▶│  ROM代码 │───▶│ Bootloader│───▶│  主程序  │
└──────────┘    └────┬─────┘    └────┬─────┘    └──────────┘
                     │               │
                     ▼               ▼
              ┌──────────┐    ┌──────────┐
              │ 验证签名 │    │ 验证签名 │
              │ Root Key │    │ App Key  │
              └────┬─────┘    └────┬─────┘
                   │               │
                   ▼               ▼
              通过: 继续      通过: 继续
              失败: 停止      失败: 进入恢复模式

信任链 (Chain of Trust):
Root of Trust ──▶ Bootloader ──▶ OS ──▶ Applications
    (硬件)          (签名校验)    (签名校验)  (签名校验)

关键技术:
├─ 公钥基础设施 (PKI)
├─ 数字签名算法 (RSA/ECC)
├─ 安全存储 (OTP/eFuse)
└─ 可信根 (Root of Trust)
```

### 4.2 硬件安全模块 (HSM/TPM)

| 模块 | 全称 | 功能 | 应用场景 |
|-----|------|------|---------|
| **TPM** | Trusted Platform Module | 密钥存储、加密运算、平台完整性 | 车辆身份认证 |
| **HSM** | Hardware Security Module | 高性能加密、密钥管理、安全协议 | V2X通信安全 |
| **TEE** | Trusted Execution Environment | 安全执行环境、敏感运算隔离 | 生物识别、支付 |
| **SE** | Secure Element | 防篡改存储、安全应用 | 数字钥匙 |

### 4.3 加密与认证

```
车载通信加密方案:

对称加密 (速度快):
├─ AES-128/256: 数据加密
├─ 用途: 大量数据传输加密
└─ 密钥管理: 通过非对称加密交换

非对称加密 (密钥交换):
├─ RSA-2048/4096: 数字签名
├─ ECC P-256: V2X标准
├─ 用途: 身份认证、密钥协商
└─ 性能: 较慢，用于密钥交换

哈希算法 (完整性):
├─ SHA-256: 数据完整性校验
├─ HMAC: 消息认证码
└─ 用途: 防篡改

V2X专用: IEEE 1609.2
├─ 椭圆曲线加密 (ECC)
├─ 隐式证书 (Implicit Certificate)
└─ 低延迟验证 (<10ms)
```

### 4.4 入侵检测系统 (IDS)

| 检测类型 | 检测内容 | 响应措施 |

| **主机IDS** | ECU行为异常 | 告警、隔离 |
| **诊断IDS** | 异常诊断请求 | 日志、阻断 |
| **物理IDS** | 异常物理接入 | 告警、限制功能 |

| 安全机制 | 技术 | 目的 |
|---------|------|------|
