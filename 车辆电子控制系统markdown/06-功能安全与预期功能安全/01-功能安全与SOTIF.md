# 功能安全与预期功能安全(SOTIF)

## 1. 功能安全基础

### 1.1 功能安全定义

功能安全是指**不存在由电子电气系统故障行为导致的不合理风险**。

```
功能安全 vs 其他安全概念:

功能安全 (Functional Safety):
├─ 标准: ISO 26262
├─ 关注点: 系统故障导致的危险
├─ 例子: 制动系统失效导致碰撞
└─ 解决: 冗余设计、故障检测

预期功能安全 (SOTIF):
├─ 标准: ISO 21448
├─ 关注点: 功能不足导致的危险
├─ 例子: 传感器无法识别白车撞白墙
└─ 解决: 性能提升、场景覆盖

网络安全 (Cyber Security):
├─ 标准: ISO/SAE 21434
├─ 关注点: 恶意攻击导致的危险
├─ 例子: 远程控制车辆
└─ 解决: 加密、认证、防火墙

三者关系: 功能安全 + SOTIF + 网络安全 = 汽车安全
```

## 2. ASIL 等级

### 2.1 ASIL定义与分级

ASIL (Automotive Safety Integrity Level) 是汽车安全完整性等级，从A到D递增。

| ASIL等级 | 风险降低要求 | 典型应用 | 硬件诊断覆盖率 | 软件要求 |
|---------|-------------|---------|---------------|---------|
| **QM** | 质量管理 | 信息娱乐 | 无 | 标准软件开发 |
| **ASIL-A** | 低 | 后视镜调节 | 60% | 适度严格 |
| **ASIL-B** | 中 | 大灯控制 | 70% | 较严格 |
| **ASIL-C** | 高 | 大灯自适应 | 80% | 严格 |
| **ASIL-D** | 最高 | 制动、转向 | 90% | 最严格 |

### 2.2 ASIL确定方法 (HARA)

```
危害分析与风险评估 (HARA):

场景: 车辆高速行驶中制动失效

严重性 (S):
├─ S0: 无伤害
├─ S1: 轻/中度伤害
├─ S2: 重度/危及生命(可能存活)
└─ S3: 危及生命(可能死亡) ← 高速碰撞: S3

暴露概率 (E):
├─ E0: 几乎不可能
├─ E1: 低概率(罕见)
├─ E2: 中概率(偶尔)
├─ E3: 高概率(经常)
└─ E4: 非常高(多数驾驶时间) ← 制动使用: E4

可控性 (C):
├─ C0: 一般可控
├─ C1: 简单可控
├─ C2: 通常可控(多数司机)
└─ C3: 难以控制/不可控 ← 高速制动失效: C3

ASIL确定:
S3 + E4 + C3 = ASIL-D (最高等级)

常见系统ASIL等级:
├─ 制动系统: ASIL-D
├─ 转向系统: ASIL-D
├─ 安全气囊: ASIL-D
├─ 发动机控制: ASIL-B/C
├─ 仪表显示: ASIL-B
└─ 信息娱乐: QM
```

### 2.3 ASIL分解原则

```
ASIL分解示例 (制动系统):

原始要求: 制动功能 ASIL-D

分解方案1 (冗余):
          制动功能 ASIL-D
                  │
      ┌───────────┴───────────┐
      │                       │
   主制动系统              冗余制动
   ASIL-D(分解为)          ASIL-D(分解为)
      │                       │
   ASIL-B(D)              ASIL-B(D)
   或 ASIL-D + QM          或 QM + ASIL-D

分解方案2 (异构冗余):
   主制动系统(液压)        备用制动(电子)
   ASIL-D(分解为)          ASIL-D(分解为)
      │                       │
   ASIL-B(D)              ASIL-B(D)
   不同技术               不同技术
   共因失效风险低

分解规则:
1. 分解后两个元素都必须满足安全要求
2. 分解后的ASIL等级可以不同
3. 需要独立性分析(避免共因失效)
4. 分解标记: ASIL-A(B)表示从B分解到A
```

## 3. 功能安全机制

### 3.1 硬件安全机制

| 机制 | 实现方式 | 覆盖故障 | ASIL适用 |
|-----|---------|---------|---------|
| **双核锁步** | 主核+从核比对 | CPU计算错误 | D |
| **ECC内存** | 纠错码 | 内存位翻转 | B-D |
| **看门狗** | 定时喂狗 | 程序跑飞 | A-D |
| **电压监控** | ADC检测 | 电源异常 | B-D |
| **时钟监控** | 双时钟比对 | 时钟失效 | B-D |
| **温度监控** | 温度传感器 | 过热 | B-D |
| **回环测试** | 数据回环 | 外设故障 | A-C |

**双核锁步架构**:
```
双核锁步 (Dual-Core Lock-Step):

┌─────────────────────────────────────────────────────────────┐
│                    安全MCU内部                               │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────┐              ┌──────────────┐            │
│  │   CPU核心0   │              │   CPU核心1   │            │
│  │   (主核)     │──────────────│   (从核)     │            │
│  │              │   输入同步    │              │            │
│  │   执行指令   │              │   执行指令   │            │
│  │              │   输出比对    │              │            │
│  └──────┬───────┘              └──────┬───────┘            │
│         │                             │                     │
│         └───────────┬─────────────────┘                     │
│                     ▼                                       │
│              ┌──────────────┐                              │
│              │   比较器     │                              │
│              │  输出对比    │                              │
│              └──────┬───────┘                              │
│                     │                                       │
│              一致───┴───不一致                               │
│                     │                                       │
│                     ▼                                       │
│              触发安全响应                                   │
└─────────────────────────────────────────────────────────────┘

延迟锁步: 核心1比核心0延迟1-2个时钟周期，
          避免共模干扰导致的同步错误
```

### 3.2 软件安全机制

| 机制 | 功能 | 实现方式 |
|-----|------|---------|
| **安全监控器** | 监控任务执行 | 独立安全任务 |
| **内存保护** | 防止越界访问 | MPU/MMU |
| **时序监控** | 检测任务超时 | 看门狗、时间监控 |
| **数据校验** | 检测数据错误 | CRC、Checksum |
| **安全通信** | E2E保护 | E2E协议 |
| **多样化软件** | 避免共因故障 | 异构算法 |

### 3.3 E2E (End-to-End) 保护

```
E2E保护通信:

发送端:
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   应用数据   │───▶│   E2E包装    │───▶│   总线发送   │
│  (有效载荷)  │    │  +Counter    │    │              │
└──────────────┘    │  +CRC        │    └──────────────┘
                    │  +Alive      │
                    └──────────────┘

接收端:
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   总线接收   │───▶│   E2E校验    │───▶│   应用数据   │
│              │    │  Counter检查 │    │  (有效载荷)  │
└──────────────┘    │  CRC验证     │    └──────────────┘
                    │  Alive检查   │
                    └──────────────┘
                    异常→安全响应

E2E Profile级别:
Profile 1: 1字节CRC, 1字节Counter - 低安全等级
Profile 2: 4字节CRC, 4字节Counter - 高安全等级
Profile 4: 8字节CRC, 2字节Counter - 最高安全等级
Profile 5: 4字节CRC, 2字节Sequence - 灵活配置
Profile 7: 4字节CRC, 2字节Counter - 时间触发
```

## 4. 软件开发流程

### 4.1 ISO 26262 软件要求

| ASIL等级 | 方法要求 | 设计验证 | 编码规范 |
|---------|---------|---------|---------|
| **QM** | 无特殊要求 | 测试 | MISRA建议 |
| **ASIL-A** | 1a | 测试+分析 | MISRA建议 |
| **ASIL-B** | 1a+1b | 测试+分析 | MISRA强制 |
| **ASIL-C** | 1a+1b+1c | 测试+分析+审查 | MISRA强制 |
| **ASIL-D** | 全部(1a-1d) | 全部方法 | MISRA强制+扩展 |

**方法1a-1d说明**:
- 1a: 基于模型的开发
- 1b: 样式指南
- 1c: 设计原则(低复杂度)
- 1d: 验证方法多样性

### 4.2 软件架构安全分析

```
软件架构安全设计:

分层架构:
┌─────────────────────────────────────────────────────────────┐
│                    应用层 (ASIL-D)                           │
│              制动控制、转向控制、安全监控                     │
├─────────────────────────────────────────────────────────────┤
│                    中间件 (ASIL-B/QM)                        │
│              通信、存储、操作系统                             │
├─────────────────────────────────────────────────────────────┤
│                    驱动层 (ASIL-B)                           │
│              外设驱动、板级支持包                             │
├─────────────────────────────────────────────────────────────┤
│                    硬件层 (ASIL-D)                           │
│              安全MCU、看门狗、电源监控                        │
└─────────────────────────────────────────────────────────────┘

分区隔离:
- 不同ASIL等级的软件分区运行
- 使用MPU隔离内存空间
- 时间保护: 高ASIL任务优先执行
- 故障传播: 低级错误不影响高ASIL
```

## 5. SOTIF (ISO 21448)

### 5.1 SOTIF定义与范围

SOTIF (Safety of the Intended Functionality) 关注**没有功能不足或合理可预见误用导致的危害**。

```
SOTIF解决的安全缺口:

功能安全 (ISO 26262)          SOTIF (ISO 21448)
     │                              │
     ▼                              ▼
┌──────────┐                ┌──────────────┐
│ 系统故障 │                │ 性能不足     │
│ 随机硬件 │                │ 传感器限制   │
│ 系统性软件│                │ 算法局限     │
│ 故障     │                │ 场景覆盖不足 │
└──────────┘                └──────────────┘
     │                              │
     └──────────┬───────────────────┘
                ▼
         汽车整体安全

SOTIF关键场景:
├─ 感知不足: 恶劣天气、特殊目标(白墙)
├─ 决策不足: 复杂交通场景
├─ 执行不足: 执行器物理限制
└─ 人机交互: 驾驶员误解系统能力
```

### 5.2 SOTIF危害场景分类

| 类别 | 描述 | 例子 | 缓解措施 |
|-----|------|------|---------|
| **Area I** | 已知安全场景 | 正常驾驶 | 无需处理 |
| **Area II** | 已知不安全场景 | 已知恶劣天气限制 | 限制使用条件 |
| **Area III** | 未知不安全场景 | 新型边缘案例 | 持续测试、ODD限制 |
| **Area IV** | 未知安全场景 | - | - |

```
SOTIF场景区域模型:

         已知场景
            │
     ┌──────┼──────┐
     │      │      │
安全 │  I   │  II  │ 不安全
     │      │      │
─────┼──────┼──────┼─────
     │      │      │
未知 │  IV  │ III  │ 不安全
场景 │      │      │
     └──────┴──────┘

目标: 扩大Area I，识别并处理Area II和III
```

### 5.3 触发条件分析与评估

| 触发条件 | 场景示例 | SOTIF措施 |
|---------|---------|----------|
| **天气** | 暴雨、大雪、浓雾 | ODD限制、降级策略 |
| **光照** | 强光直射、黑夜 | 传感器冗余、HDR |
| **道路** | 无标线、施工区 | 高精地图、定位增强 |
| **目标** | 异形车辆、动物 | 训练数据增强 |
| **行为** | 加塞、逆行 | 预测模型优化 |

### 5.4 ODD (设计运行域)

ODD定义了系统设计的运行条件边界。

```
ODD定义要素:

环境条件:
├─ 天气: 晴天、多云、小雨
├─ 光照: 白天、黄昏、有路灯夜间
├─ 温度: -20°C ~ +50°C
└─ 海拔: <3000m

道路条件:
├─ 类型: 高速公路、城市快速路
├─ 标线: 清晰可见
├─ 路面: 干燥、柏油/水泥
└─ 坡度: <±15%

交通条件:
├─ 车速: 0-120km/h
├─ 密度: 正常交通流
└─ 参与者: 常规车辆、行人

操作限制:
├─ 驾驶员: 手握方向盘，随时接管
├─ 时间: 单次最长30分钟
└─ 地理: 高精地图覆盖区域

超出ODD → 系统退出或降级
```
